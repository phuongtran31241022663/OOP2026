State Pattern trong C#


1. Khái niệm và động lực


State Pattern là một mẫu thiết kế hành vi (Behavioral Pattern) cho phép một đối tượng thay đổi hành vi khi trạng thái bên trong của nó thay đổi. Nhìn từ bên ngoài, dường như đối tượng đó đã thay đổi lớp.


Bài toán điển hình: một đối tượng có một tập hữu hạn các trạng thái, mỗi trạng thái ứng xử khác nhau với cùng một sự kiện. Nếu dùng cấu trúc switch-case hay if-else để kiểm tra trạng thái, mã nguồn sẽ:


· Bị trùng lặp logic kiểm tra trạng thái ở nhiều phương thức.
· Vi phạm nguyên tắc Open/Closed vì mỗi khi thêm trạng thái mới, ta phải sửa tất cả các switch-case.
· Khó kiểm thử từng trạng thái độc lập.


State Pattern giải quyết bằng cách đóng gói hành vi của mỗi trạng thái vào một lớp riêng, và ủy thác (delegate) hành vi từ context hiện tại sang đối tượng trạng thái.


---


2. Cấu trúc cổ điển trong C#


Mẫu bao gồm ba thành phần chính:


· Context: Lớp mà client quan tâm, duy trì một tham chiếu đến State hiện tại. Nó công khai các phương thức đại diện cho hành vi, nhưng thực chất chỉ gọi state.Handle(this).
· State: Một abstract class hoặc interface khai báo các hành vi phụ thuộc trạng thái.
· ConcreteState: Các lớp cụ thể triển khai từng trạng thái. Mỗi lớp tự quyết định chuyển đổi sang trạng thái khác bằng cách gán context.State = new OtherState().


Ví dụ: Máy bán vé tự động


```csharp
// Interface trạng thái
public interface IState
{
    void InsertCoin();
    void SelectTicket();
    void Cancel();
}


// Context
public class TicketMachine
{
    private IState _currentState;


    public TicketMachine()
    {
        // Khởi tạo máy ở trạng thái chờ đồng xu
        _currentState = new WaitingForCoinState(this);
    }


    public IState CurrentState
    {
        get => _currentState;
        set => _currentState = value;
    }


    public void InsertCoin() => _currentState.InsertCoin();
    public void SelectTicket() => _currentState.SelectTicket();
    public void Cancel() => _currentState.Cancel();
}


// Các ConcreteState
public class WaitingForCoinState : IState
{
    private readonly TicketMachine _machine;


    public WaitingForCoinState(TicketMachine machine) => _machine = machine;


    public void InsertCoin()
    {
        Console.WriteLine("Coin accepted.");
        _machine.CurrentState = new TicketSelectedState(_machine);
    }


    public void SelectTicket() => Console.WriteLine("Insert a coin first.");
    public void Cancel() => Console.WriteLine("No coin to return.");
}


public class TicketSelectedState : IState
{
    private readonly TicketMachine _machine;


    public TicketSelectedState(TicketMachine machine) => _machine = machine;


    public void InsertCoin() => Console.WriteLine("Coin already inserted.");
    public void SelectTicket()
    {
        Console.WriteLine("Printing ticket...");
        _machine.CurrentState = new WaitingForCoinState(_machine);
    }
    public void Cancel()
    {
        Console.WriteLine("Refunding coin...");
        _machine.CurrentState = new WaitingForCoinState(_machine);
    }
}
```


Như vậy, logic switch(state) biến mất khỏi TicketMachine. Mỗi trạng thái đều đóng gói hành vi của nó và tự quản lý chuyển đổi.

3. Các biến thể nâng cao


3.1 State Machine phân cấp (Hierarchical State)


Khi trạng thái có các trạng thái con (ví dụ: trạng thái "Đang hoạt động" bao gồm "Chạy", "Đi bộ"), ta có thể áp dụng Composite + State. Một lớp trạng thái có thể duy trì một state machine con. C# không có hỗ trợ built-in nhưng có thể xây dựng bằng tay, tuy nhiên các thư viện như Stateless rất phổ biến cho việc này.


Ví dụ dùng thư viện Stateless (nếu cần giới thiệu):


```csharp
var machine = new StateMachine<Trigger, State>(State.Idle);


machine.Configure(State.Idle)
    .Permit(Trigger.Start, State.Active);


machine.Configure(State.Active)
    .Permit(Trigger.Stop, State.Idle)
    .OnEntry(() => Console.WriteLine("Machine active"));
```


Dù có thể dùng thư viện, việc tự code State Pattern giúp bạn kiểm soát hoàn toàn và tận dụng tối đa tính năng ngôn ngữ.


3.2 State và Dependency Injection


Khi các state cần phụ thuộc vào service bên ngoài (logging, repository), việc khởi tạo new ConcreteState(...) trong context truyền thống có thể phá vỡ DI. Giải pháp:


· Dùng Factory cho state, inject IStateFactory vào context.
· Đăng ký states như các dịch vụ transient, và context nhận IServiceProvider để resolve mỗi khi cần chuyển trạng thái.
· Sử dụng Event-driven state: context phát ra sự kiện, các state tự đăng ký, không cần context biết cụ thể class nào.


3.3 Thread Safety


Trong môi trường đa luồng, việc chuyển đổi state không nguyên tử (read current, create new, assign) có thể gây race condition. Có thể:


· Sử dụng Interlocked.Exchange nếu state là reference type.
· Dùng ImmutableState + context bất biến (với record) là an toàn tuyệt đối.
· Sử dụng khóa (lock) nhưng cần cẩn thận với deadlock và hiệu năng.

4. So sánh với Strategy Pattern


Nhiều lập trình viên nhầm lẫn State và Strategy vì cả hai đều dùng composition để thay đổi hành vi. Khác biệt chính:


· Strategy: client chọn một strategy từ bên ngoài và thường không thay đổi trong suốt vòng đời. Strategy không tự chuyển đổi lẫn nhau.
· State: trạng thái thay đổi từ bên trong, mỗi state biết về state khác và tự quyết định khi nào chuyển đổi. Đây là điểm mấu chốt: các concrete state thường biết đến ít nhất một state khác (hoặc context).


Trong C#, Strategy hay dùng delegate/Action, còn State luôn liên quan đến một vòng đời và các lớp riêng biệt.


---


5. Ứng dụng thực tế


· Vòng đời đơn hàng: New → Confirmed → Shipped → Delivered → Cancelled, mỗi trạng thái chỉ cho phép các chuyển đổi nhất định.
· UI Navigation / Workflow: Các bước trong wizard, form đăng ký, luồng thanh toán.
· Trình phát nhạc: Playing, Paused, Stopped.
· Game development: trạng thái nhân vật, trạng thái game.
· Async/await trình biên dịch: Bản thân compiler của C# sinh ra state machine cho async method, dùng IAsyncStateMachine và AsyncVoidMethodBuilder – một ví dụ tuyệt vời của pattern này được áp dụng trong ngôn ngữ.


---


6. Lưu ý và thực tiễn triển khai


· Phạm vi: Nếu chỉ có 2-3 trạng thái và ít hành vi, có thể dùng enum + switch để tránh quá tải kỹ thuật. State Pattern phát huy lợi thế khi số lượng trạng thái và hành vi tăng, cần mở rộng sau này.
· Quản lý chuyển đổi: Nên tập trung logic chuyển đổi vào context hoặc một lớp riêng (transition table) thay vì để mỗi state tự gán context.State làm tăng coupling. Một cách hay là dùng sự kiện trả về State mới, context thực hiện thay đổi.
· Serialization: Khi serialize context, cần cả trạng thái hiện tại. Dùng [JsonDerivedType] cho State abstract để deserialize đúng kiểu. Nếu dùng record, serialization trở nên tự nhiên.
· Kiểm thử: Mỗi concrete state dễ dàng kiểm thử đơn vị (unit test). Có thể mock context hoặc chỉ kiểm tra hành vi của state khi nhận sự kiện.


---


7. Kết luận


State Pattern là công cụ mạnh mẽ để thay thế các khối điều kiện phức tạp bằng một tập các lớp có trách nhiệm rõ ràng, tuân thủ nguyên lý Single Responsibility và Open/Closed. Trong C#, pattern này hòa quyện với các tính năng hiện đại như record, pattern matching, và default interface methods để tạo ra code sạch, bất biến, an toàn luồng và dễ kiểm thử. Khi thiết kế hệ thống hướng đối tượng, việc nhận diện các đối tượng có vòng đời hành vi thay đổi và áp dụng State Pattern sẽ giúp mã nguồn linh hoạt hơn trước những yêu cầu thay đổi trong tương lai.
