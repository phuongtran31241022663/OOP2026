Strategy Pattern trong C#


1. Khái niệm và mục đích


Strategy Pattern (mẫu chiến lược) là một mẫu thiết kế hành vi cho phép định nghĩa một họ các thuật toán, đóng gói mỗi thuật toán vào một lớp riêng biệt và làm cho chúng có thể hoán đổi cho nhau tại thời điểm chạy. Strategy giúp thay đổi hành vi của một đối tượng một cách độc lập với client sử dụng đối tượng đó.


Mục tiêu cốt lõi của Strategy là:


· Tách biệt phần thay đổi (thuật toán) ra khỏi phần ổn định (context).
· Tuân thủ nguyên lý Open/Closed: có thể mở rộng bằng cách thêm strategy mới mà không sửa đổi context hay các strategy hiện có.
· Tránh các khối switch-case hoặc if-else dài dựa trên kiểu hay cờ quyết định hành vi, giúp mã sạch, dễ bảo trì hơn.


Ví dụ kinh điển: một ứng dụng tính phí vận chuyển, nơi thuật toán tính phí thay đổi theo nhà vận chuyển (UPS, FedEx, VNPost...). Thay vì nhồi tất cả công thức vào một phương thức với điều kiện rẽ nhánh, Strategy cho phép mỗi nhà vận chuyển có một lớp riêng, và client chỉ cần chọn một chiến lược phù hợp.

2. Cấu trúc cổ điển trong C#


Dựa trên mẫu GoF, Strategy Pattern gồm ba vai trò:


· Strategy: Interface hoặc abstract class khai báo phương thức thực thi thuật toán.
· ConcreteStrategy: Các lớp cụ thể triển khai interface Strategy, mỗi lớp một thuật toán.
· Context: Lớp sử dụng Strategy, duy trì một tham chiếu đến Strategy và ủy thác công việc cho nó.


Ví dụ: Tính phí vận chuyển


```csharp
// 1. Strategy interface
public interface IShippingStrategy
{
    decimal CalculateCost(decimal weight, string origin, string destination);
}


// 2. Concrete strategies
public class UpsShipping : IShippingStrategy
{
    public decimal CalculateCost(decimal weight, string origin, string destination)
    {
        // Giả lập thuật toán của UPS
        return weight * 2.5m + 10;
    }
}


public class FedExShipping : IShippingStrategy
{
    public decimal CalculateCost(decimal weight, string origin, string destination)
    {
        return weight * 3.0m + 8;
    }
}


public class VnPostShipping : IShippingStrategy
{
    public decimal CalculateCost(decimal weight, string origin, string destination)
    {
        return weight * 1.8m + (destination.StartsWith("Hanoi") ? 5 : 12);
    }
}


// 3. Context
public class ShippingCalculator
{
    // Strategy được inject từ bên ngoài (constructor injection)
    private readonly IShippingStrategy _strategy;


    public ShippingCalculator(IShippingStrategy strategy)
    {
        _strategy = strategy ?? throw new ArgumentNullException(nameof(strategy));
    }


    public decimal Calculate(decimal weight, string origin, string destination)
        => _strategy.CalculateCost(weight, origin, destination);
}
```


Sử dụng:


```csharp
// Client chọn chiến lược
IShippingStrategy strategy = new UpsShipping(); // Có thể lấy từ DI container, factory...
var calculator = new ShippingCalculator(strategy);
decimal cost = calculator.Calculate(2.5m, "HCMC", "Hanoi");
Console.WriteLine($"Shipping cost: {cost}");
```


Nhận xét: Cấu trúc rất rõ ràng, dễ kiểm thử từng strategy độc lập, dễ thêm mới chiến lược mà không cần sửa ShippingCalculator.

3. Sự tiến hóa với C# hiện đại


Các tính năng hiện đại của C# giúp giảm đáng kể boilerplate code và mang đến những cách triển khai linh hoạt hơn, phù hợp với từng tình huống.


3.1 Dùng delegate/Action/Func thay cho Interface riêng


Khi strategy chỉ có một phương thức duy nhất, việc tạo interface và nhiều lớp cụ thể trở nên nặng nề. C# cho phép sử dụng delegate Func<> hoặc Action<> để đại diện trực tiếp cho thuật toán.


```csharp
public class ShippingCalculator
{
    private readonly Func<decimal, string, string, decimal> _calculateFunc;


    public ShippingCalculator(Func<decimal, string, string, decimal> calculateFunc)
        => _calculateFunc = calculateFunc;


    public decimal Calculate(decimal weight, string origin, string dest)
        => _calculateFunc(weight, origin, dest);
}
```


Sử dụng:


```csharp
// Định nghĩa các chiến lược như lambda hoặc phương thức tĩnh
var upsStrategy = (decimal w, string o, string d) => w * 2.5m + 10;
var fedexStrategy = (decimal w, string o, string d) => w * 3.0m + 8;


var calculator = new ShippingCalculator(upsStrategy);
decimal cost = calculator.Calculate(2.5m, "HCMC", "Hanoi");
```


Cách này cực kỳ nhẹ, phù hợp khi các thuật toán ngắn gọn và không có trạng thái riêng. Nhược điểm: mất đi ý nghĩa tường minh của từng strategy (không có tên lớp), khó tái sử dụng nếu logic phức tạp.


3.2 Strategy với Generic và Constraint


Có thể tham số hóa Context bằng kiểu Strategy, ràng buộc nó phải triển khai interface:


```csharp
public class ShippingCalculator<TStrategy> where TStrategy : IShippingStrategy, new()
{
    private readonly TStrategy _strategy = new();


    public decimal Calculate(decimal weight, string origin, string dest)
        => _strategy.CalculateCost(weight, origin, dest);
}
// Sử dụng: new ShippingCalculator<UpsShipping>().Calculate(...)
```


Hoặc dùng với dependency injection để tự động resolve.


3.3 Strategy với Dependency Injection


Trong các ứng dụng .NET hiện đại, ta thường đăng ký các strategy vào DI container, và context nhận chúng qua constructor, hoặc dùng IEnumerable<T> để lấy tất cả, hoặc sử dụng factory.


Đăng ký:


```csharp
services.AddTransient<IShippingStrategy, UpsShipping>();
services.AddTransient<IShippingStrategy, FedExShipping>();
services.AddTransient<IShippingStrategy, VnPostShipping>();
```


Context sử dụng factory để chọn strategy:


```csharp
public class ShippingCalculator
{
    private readonly IEnumerable<IShippingStrategy> _strategies;


    public ShippingCalculator(IEnumerable<IShippingStrategy> strategies) => _strategies = strategies;


    public decimal Calculate(string carrier, decimal weight, string origin, string dest)
    {
        var strategy = _strategies.FirstOrDefault(s => s.CarrierName == carrier);
        if (strategy == null) throw new ArgumentException($"Unknown carrier: {carrier}");
        return strategy.CalculateCost(weight, origin, dest);
    }
}
```


3.4 Strategy như record hoặc bất biến


Khi strategy chỉ là một hàm nhưng vẫn muốn có tên gọi, có thể dùng record để đóng gói delegate:


```csharp
public record ShippingStrategy(Func<decimal, string, string, decimal> CalculateFunc);
```


3.5 Pattern matching để lựa chọn strategy


Đôi khi việc tạo lớp cụ thể là không cần thiết nếu ta có một enum hoặc union type. C# cho phép kết hợp pattern matching với record và switch expression để viết ngắn gọn:


```csharp
public record Order(decimal Amount, CustomerTier Tier);


public decimal CalculateDiscount(Order order) => order.Tier switch
{
    CustomerTier.Regular => order.Amount * 0.05m,
    CustomerTier.Premium => order.Amount * 0.10m,
    CustomerTier.VIP => order.Amount * 0.15m,
    _ => 0
};
```


Tuy không phải Strategy Pattern chuẩn (vì không có đối tượng strategy riêng) nhưng vẫn thể hiện triết lý tương tự: tách biệt họ thuật toán dựa trên một discriminator, và dễ dàng mở rộng (thêm tier mới sẽ phải sửa switch, vi phạm Open/Closed – phù hợp khi tập tier nhỏ và ổn định).

4. So sánh với các mẫu liên quan


4.1 Strategy vs State


Cả hai đều sử dụng composition và interface để thay đổi hành vi. Khác biệt chính:


· Strategy: client (hoặc context) chọn chiến lược từ bên ngoài và thường không thay đổi trong suốt vòng đời (hoặc thay đổi theo ý muốn tường minh của client). Các strategy không biết về nhau và không tự động chuyển đổi.
· State: trạng thái thay đổi từ bên trong đối tượng, mỗi state có thể biết về các state khác và tự quyết định khi nào cần chuyển đổi. Đây là điểm khác biệt then chốt.


Trong C#, State thường yêu cầu quản lý vòng đời phức tạp, trong khi Strategy đơn giản hơn do tính một chiều.


4.2 Strategy vs Template Method


· Template Method dùng kế thừa (abstract class) để định nghĩa khung thuật toán, các bước cụ thể được ghi đè bởi lớp con. Cùng một kế thừa, chỉ một biến thể tại một thời điểm.
· Strategy dùng ủy thác (delegation) cho phép thay đổi toàn bộ thuật toán tại runtime mà không cần kế thừa context. Strategy linh hoạt hơn, tuân thủ ưu tiên composition over inheritance.


Trong C#, Template Method vẫn có chỗ đứng khi thuật toán có cấu trúc chung và muốn chia sẻ mã dùng chung. Strategy thường được ưa chuộng hơn nhờ khả năng tiêm phụ thuộc và tách biệt.

5. Các biến thể nâng cao


5.1 Strategy Factory


Để tránh cho context phải biết quá nhiều về cách chọn strategy, ta có thể tạo một IStrategyFactory để tạo ra strategy thích hợp dựa trên tham số.


```csharp
public interface IShippingStrategyFactory
{
    IShippingStrategy Create(string carrierName);
}
```


Factory này có thể sử dụng dictionary với key là tên carrier, giúp tra cứu nhanh, hoặc dùng DI container.


5.2 Strategy với Default Implementation trong Interface


Từ C# 8, interface cho phép default implementation. Điều này cho phép cung cấp hành vi mặc định cho tất cả strategy mà không cần abstract class.


```csharp
public interface IShippingStrategy
{
    decimal CalculateCost(decimal weight, string origin, string dest);
    // Phương thức tiện ích có thể dùng chung
    decimal CalculateInsurance(decimal value) => value * 0.01m; // mặc định 1%
}
```


5.3 Policy-based Design với Source Generator


Ứng dụng hiện đại có thể dùng source generator để tạo ra các strategy cụ thể từ attribute hoặc convention, giảm thiểu boilerplate. Ví dụ: tạo sẵn các validator strategy cho các class DTO.


```csharp
// Imaginary Generator: [Strategy("UPS")] -> tự sinh class UpsShipping : IShippingStrategy
```


5.4 Strategy với Ardalis.Specification hoặc GuardClauses


Trong các hệ thống phức tạp, strategy có thể được dùng để kiểm tra điều kiện logic (specification pattern) hay xác thực.

6. Ứng dụng thực tế trong .NET


6.1 Sắp xếp với IComparer<T>


Framework .NET đã sử dụng Strategy Pattern thông qua IComparer<T>. Bạn có thể hoán đổi cách sắp xếp danh sách bằng cách truyền một comparer khác.


```csharp
var people = new List<Person> { ... };
people.Sort((p1, p2) => p1.Age.CompareTo(p2.Age)); // lambda delegate
people.Sort(new AgeComparer()); // object strategy
```


LINQ OrderBy cũng nhận delegate (Func<TSource, TKey>) làm strategy trích xuất khóa.


6.2 Xác thực (Validation)


FluentValidation sử dụng Strategy tương tự: mỗi validator là một strategy cho một kiểu dữ liệu cụ thể.


6.3 Gửi email (SmptClient, SendGrid, MailChimp)


Interface IEmailService với nhiều implementation cho các nhà cung cấp khác nhau. Đây là Strategy điển hình.


6.4 Thanh toán (Payment Gateway)


IPaymentGateway với các class PaypalPayment, StripePayment, MomoPayment. Context PaymentProcessor nhận strategy và xử lý.


6.5 Thuật toán nén (Compression)


ICompressionStrategy có các lớp GZipCompression, BrotliCompression, NoCompression.


6.6 ASP.NET Core: Middleware pipeline


Bản thân pipeline không phải Strategy thuần túy, nhưng mỗi middleware có thể coi là một strategy được chọn lựa dựa trên đường dẫn, điều kiện. Đặc biệt, UseWhen và MapWhen nhận predicate để quyết định branch.

7. Lưu ý và best practices


1. Ưu tiên delegate cho strategy đơn giản: Nếu strategy chỉ là một hàm duy nhất, không có trạng thái, hãy dùng Func<> hoặc Action<> thay vì tạo interface và class riêng. Lợi: giảm mã, linh hoạt. Nhược: giảm khả năng đọc (thiếu tên gọi rõ ràng). Có thể khắc phục bằng cách đóng gói delegate trong record có tên.
2. Đừng lạm dụng Strategy khi hành vi ít thay đổi: Nếu chỉ có hai lựa chọn, đôi khi if-else vẫn đủ. Việc tạo quá nhiều class có thể gây ra over-engineering. Hãy nhìn vào dấu hiệu mở rộng trong tương lai.
3. Quản lý vòng đời strategy đúng cách: Nếu strategy không có trạng thái, nên đăng ký chúng là Singleton trong DI để tối ưu bộ nhớ. Nếu có trạng thái (hiếm gặp), cần Scoped hoặc Transient.
4. Đặt tên rõ ràng: Tên strategy nên thể hiện thuật toán cụ thể (ví dụ: UpsShipping, FedExShipping) để dễ đọc và bảo trì.
5. Kết hợp với Factory để chọn strategy: Context không nên chứa logic phức tạp để chọn strategy. Sử dụng Factory hoặc DI container để quyết định, giữ cho context tập trung vào nhiệm vụ chính.
6. Kiểm tra null và strategy mặc định: Nếu không có strategy nào được cung cấp, hãy có fallback (Null Object Pattern) thay vì để trống.
7. Không để strategy biết quá nhiều về context: Strategy nên nhận đủ dữ liệu qua tham số, tránh phụ thuộc ngược vào context (trừ khi thực sự cần thiết).
8. Strategy có thể sử dụng dữ liệu cấu hình: Có thể inject IOptions<T> vào strategy để đọc cấu hình mà không cần hardcode.
9. Kết hợp Strategy với Template Method: Khi muốn có một khung xử lý chung nhưng từng bước có thể thay đổi được. Điều này tạo ra sự phân cấp linh hoạt.


---


8. Kết luận


Strategy Pattern trong C# đã phát triển vượt xa mẫu thiết kế nguyên thủy nhờ các tính năng ngôn ngữ mạnh mẽ. Từ việc phải tạo interface và nhiều lớp cụ thể, nay ta có thể dùng delegate, generic, dependency injection và record để triển khai pattern này một cách súc tích và an toàn. Sự kết hợp giữa Strategy và các nguyên lý như composition over inheritance, DI, Open/Closed đã khiến nó trở thành một trong những mẫu linh hoạt nhất, hiện diện khắp nơi trong hệ sinh thái .NET, từ sắp xếp, xác thực, thanh toán đến pipeline xử lý HTTP. Nắm vững Strategy cũng là nắm vững cách tách biệt cái thay đổi khỏi cái ổn định, một kỹ năng cốt lõi của lập trình hướng đối tượng chất lượng cao.
