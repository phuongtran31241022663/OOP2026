Tính đóng gói (Encapsulation) trong C#


Trong mô hình Lập trình hướng đối tượng (OOP), đóng gói không đơn thuần chỉ là ẩn dữ liệu – đó là nghệ thuật kiểm soát truy cập và bảo vệ tính toàn vẹn trạng thái của một đối tượng. C# cung cấp một bộ công cụ vô cùng phong phú để thực hiện đóng gói, vượt xa những gì thường thấy trong các ngôn ngữ OOP truyền thống.


---


1. Bản chất của đóng gói – Hơn cả ẩn dữ liệu


Đóng gói là cơ chế gói gọn dữ liệu (fields) và phương thức (methods) hoạt động trên dữ liệu đó vào một đơn vị duy nhất (class/struct), đồng thời hạn chế quyền truy cập trực tiếp từ bên ngoài. Mục tiêu cốt lõi gồm:


· Bảo vệ tính toàn vẹn dữ liệu: Ngăn chặn trạng thái không hợp lệ.
· Che giấu chi tiết triển khai: Cho phép thay đổi cấu trúc nội bộ mà không ảnh hưởng đến mã khách hàng.
· Kiểm soát tương tác: Quy định rõ ràng các "cửa" (phương thức, thuộc tính) mà bên ngoài có thể sử dụng.
· Hỗ trợ bảo trì và mở rộng: Giảm liên kết (coupling) giữa các phần của hệ thống.


Trong C#, đóng gói được hiện thực qua Access Modifiers, Properties, Indexers, Partial Classes, cùng các tính năng init-only, record, và các quy ước đặt tên.


---


2. Bộ công cụ đóng gói trong C#


2.1 Access Modifiers – Tường rào kiểm soát truy cập


C# cung cấp 6 mức độ truy cập, linh hoạt đến từng loại quan hệ:


Modifier Truy cập từ cùng class Từ class dẫn xuất (cùng assembly) Từ class dẫn xuất (khác assembly) Từ code không kế thừa (cùng assembly) Từ code không kế thừa (khác assembly)
public ✓ ✓ ✓ ✓ ✓
protected internal ✓ ✓ ✓ (nếu dẫn xuất) ✓ ✗
internal ✓ ✓ ✗ ✓ ✗
protected ✓ ✓ ✓ ✗ ✗
private protected ✓ ✓ (chỉ khi cùng assembly) ✗ ✗ ✗
private ✓ ✗ ✗ ✗ ✗


Ví dụ thực tiễn về private protected (từ C# 7.2):


```csharp
public class BaseClass
{
    private protected int _secretValue = 42; // Chỉ dành cho dẫn xuất *trong cùng assembly*
}


// Cùng assembly
public class DerivedSameAssembly : BaseClass
{
    public int GetSecret() => _secretValue; // Hợp lệ
}


// Ở assembly khác, dù kế thừa cũng không thấy _secretValue
```


Điều này cho phép tạo ra những ràng buộc cực kỳ chặt chẽ, chỉ chia sẻ nội bộ trong assembly cho mục đích kế thừa.


2.2 Properties – Cánh cửa đóng gói đầy quyền năng


Properties là cơ chế chính để lộ dữ liệu một cách an toàn. Chúng trông như field nhưng thực chất là phương thức (getter/setter), cho phép thêm logic kiểm tra.


```csharp
public class Account
{
    private decimal _balance;
    public decimal Balance
    {
        get => _balance;
        private set
        {
            if (value < 0)
                throw new ArgumentException("Balance cannot be negative");
            _balance = value;
        }
    }


    public void Deposit(decimal amount) => Balance += amount;
}
```


· Auto-properties ({ get; set; }) tự sinh field ẩn, giảm boilerplate.
· Expression-bodied properties (=>) làm code sạch sẽ.


2.3 Indexers – Đóng gói dạng chỉ mục


Indexers cho phép truy cập đối tượng như một mảng nhưng vẫn giữ được sự kiểm soát bên trong. Chúng có thể có bất kỳ kiểu tham số nào, không chỉ int.


```csharp
public class Week
{
    private readonly string[] _days = { "Mon", "Tue", "Wed", "Thu", "Fri", "Sat", "Sun" };
    public string this[int index]
    {
        get => (index >= 0 && index < _days.Length) ? _days[index] : throw new IndexOutOfRangeException();
        // Có thể thêm logic set nếu cần
    }
}
```


Indexers có thể được overload theo kiểu chỉ mục khác nhau, và cũng hỗ trợ init cho mục đích khởi tạo.


2.4 Partial Classes – Phân tán định nghĩa nhưng vẫn đóng gói


Partial class cho phép chia nhỏ định nghĩa của một lớp ra nhiều file, thường dùng khi kết hợp code sinh tự động (designer) và code thủ công. Dù ở nhiều file, chúng vẫn thuộc về cùng một đơn vị đóng gói và có quyền truy cập đầy đủ các thành viên private của nhau. Đây là cách C# hỗ trợ đóng gói trong bối cảnh tool-generated code mà không làm hỏng nguyên tắc OOP.


2.5 Internal và InternalsVisibleTo – Đóng gói ở cấp assembly


Mặc định, các kiểu/internal member chỉ hiển thị trong cùng assembly. Khi cần test hoặc chia sẻ giữa các assembly thân thiết, C# cho phép dùng [InternalsVisibleTo]:


```csharp
[assembly: InternalsVisibleTo("MyApp.Tests")]
```


Điều này vừa giữ đóng gói, vừa linh hoạt cho kiểm thử.

3. Đóng gói và các tính năng hiện đại hướng tới bất biến


Lập trình hướng đối tượng hiện đại ngày càng đề cao tính bất biến để tăng an toàn luồng và giảm lỗi. C# cung cấp những công cụ đóng gói giúp tạo kiểu bất biến một cách thanh lịch:


· readonly fields / readonly struct: Các trường chỉ gán được trong constructor hoặc lúc khai báo. Kể từ C# 8, readonly struct đảm bảo toàn bộ struct là bất biến.
Đóng gói ở đây thể hiện ở chỗ bạn không thể thay đổi trực tiếp thuộc tính từ bên ngoài, nhưng vẫn có cơ chế tạo bản sao "đã sửa đổi" một cách an toàn.


---


4. Đóng gói trong các mối quan hệ OOP: Kế thừa, Đa hình


Đóng gói kết hợp với kế thừa dễ gây ra những lỗ hổng như lộ chi tiết cho lớp dẫn xuất, hoặc vi phạm Liskov Substitution Principle. C# giải quyết bằng:


· virtual / override / sealed: Kiểm soát cách ghi đè. Bạn có thể ngăn chặn ghi đè tiếp bằng sealed override.
· new keyword: Ẩn thành viên lớp cơ sở một cách tường minh (không khuyến khích lạm dụng).
· protected vs private protected: Bạn có thể hạn chế truy cập từ lớp dẫn xuất ở assembly khác.
· Abstract vs Interface: Abstract class cho phép chia sẻ trạng thái (fields) và có constructor, do đó mức độ đóng gói cao hơn Interface thuần túy (vốn không có state).


Khi thiết kế, nguyên tắc "Tell, Don't Ask" khuyến khích ta không lấy dữ liệu ra để xử lý bên ngoài, mà hãy bảo đối tượng thực hiện hành vi – chính là tận dụng đóng gói để giữ logic bên trong.


---


5. Đóng gói trong Serialization – Thách thức và giải pháp


Khi serialize đối tượng, đóng gói phải đối mặt với việc đọc/ghi dữ liệu riêng tư. Các serializer hiện đại như System.Text.Json hay Newtonsoft.Json cung cấp các tùy chọn để tương tác với đóng gói:


· Public properties: Serializer mặc định chỉ làm việc với public getter/setter. Nếu setter là private, bạn cần [JsonInclude] hoặc JsonSerializerOptions.IncludeFields = true (đối với fields).
· [JsonConstructor]: Gắn constructor có tham số để deserialize bất biến (record). Điều này tôn trọng đóng gói vì ta không cần setter public.
· [JsonIgnore]: Loại trừ thuộc tính không muốn phơi bày.
Như vậy, đóng gói và serialization không đối nghịch, mà C# cho phép chúng cùng tồn tại linh hoạt.

6. Đóng gói trong các Design Pattern


Trong các pattern bạn đã tìm hiểu:


· Strategy: Mỗi strategy là một đối tượng riêng có đóng gói thuật toán, che giấu cách tính.
· State: Mỗi concrete state đóng gói hành vi của trạng thái và thường chỉ lộ hành vi qua interface chung.
· Observer: Subject đóng gói danh sách observer và logic thông báo, observer đóng gói hành vi phản ứng.
· Singleton: Sử dụng đóng gói để ẩn constructor và quản lý instance duy nhất.


Tính đóng gói là nền tảng của những pattern này, giúp thay đổi hành vi mà không làm ảnh hưởng đến phần còn lại.

7. Best Practices và các lỗi thường gặp


· Luôn dùng properties thay vì public fields. Điều này giữ sự nhất quán và cho phép thêm logic sau mà không phá vỡ tương thích nhị phân.
· Không nên lộ List<T> dạng property chỉ đọc reference: Người dùng có thể gọi .Add() làm thay đổi nội dung. Thay vào đó, dùng IReadOnlyList<T> hoặc trả về copy.
· Tránh lạm dụng protected: Nó tạo liên kết chặt giữa lớp cơ sở và dẫn xuất. Hãy ưu tiên private protected khi có thể.
· Sử dụng sealed cho class không được thiết kế để kế thừa – vừa cải thiện hiệu năng (devirtualization), vừa bảo vệ đóng gói.
· Khi làm việc với serialization, cẩn thận với [JsonInclude] trên private fields: Dễ vô tình phơi bày dữ liệu nhạy cảm.
· Tận dụng required để đảm bảo object luôn ở trạng thái hợp lệ ngay sau khởi tạo.
