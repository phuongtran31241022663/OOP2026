Observer Pattern trong C#


1. Khái niệm và mục đích


Observer Pattern (mẫu quan sát) là một mẫu thiết kế hành vi, thiết lập cơ chế phụ thuộc một-nhiều giữa các đối tượng: khi một đối tượng (gọi là Subject hay Publisher) thay đổi trạng thái, tất cả các đối tượng phụ thuộc vào nó (Observer hay Subscriber) tự động được thông báo và cập nhật. Pattern này chính là nền tảng cho kiến trúc hướng sự kiện, lập trình reactive, và tương tác UI.


Trong C#, Observer Pattern không chỉ được xây dựng thủ công theo phong cách "classic" mà còn được hỗ trợ trực tiếp qua hai cơ chế cốt lõi của ngôn ngữ:


· Delegate và Event (có từ C# 1.0).
· Giao diện IObserver<T> / IObservable<T> (giới thiệu từ .NET Framework 4.0, đặc biệt mạnh mẽ cùng Reactive Extensions).


C# cung cấp cả hai để đáp ứng các mức độ phức tạp khác nhau, từ lập trình giao diện đơn giản đến xử lý luồng dữ liệu bất đồng bộ.

2. Triển khai Observer cổ điển (GoF)


Trước khi bàn đến các tính năng có sẵn, ta nhắc lại cấu trúc gốc để thấy sự ánh xạ vào ngôn ngữ C#.


Các vai trò:


· ISubject – giao diện đăng ký, hủy đăng ký, thông báo.
· IObserver – giao diện nhận cập nhật.
· ConcreteSubject – lưu danh sách observer, khi trạng thái thay đổi gọi Notify().
· ConcreteObserver – xử lý thông báo.


Ví dụ kinh điển trong C#:


```csharp
public interface IObserver
{
    void Update(string message);
}


public interface ISubject
{
    void Attach(IObserver observer);
    void Detach(IObserver observer);
    void Notify();
}


public class NewsPublisher : ISubject
{
    private readonly List<IObserver> _observers = new();
    private string _latestNews;


    public void Attach(IObserver observer) => _observers.Add(observer);
    public void Detach(IObserver observer) => _observers.Remove(observer);


    public void Notify()
    {
        foreach (var obs in _observers)
            obs.Update(_latestNews);
    }


    public void Publish(string news)
    {
        _latestNews = news;
        Notify();
    }
}


public class Subscriber : IObserver
{
    public string Name { get; set; }
    public void Update(string message) => Console.WriteLine($"{Name} received: {message}");
}
```


Tuy rõ ràng, cách này khiến Subject phải tự duy trì danh sách, xử lý đồng bộ, và chưa tận dụng được sức mạnh của .NET.

3. Delegate và Event – Cơ chế Observer được tích hợp sẵn


C# coi phương thức là đối tượng (first-class citizen) thông qua delegate, và từ khóa event biến delegate thành một cơ chế publish-subscribe đóng gói, an toàn kiểu.


3.1 Nguyên lý


· Delegate (hành động) đại diện cho một hoặc nhiều phương thức có cùng chữ ký.
· Event là một thành viên của lớp, che giấu delegate bên trong, chỉ cho phép subscriber đăng ký (+=) hoặc hủy đăng ký (-=), không cho gán lại hay gọi trực tiếp từ ngoài.


Ví dụ đơn giản với EventHandler:


```csharp
public class Thermometer
{
    private decimal _temperature;


    // Khai báo sự kiện dùng EventHandler<T> - chuẩn .NET
    public event EventHandler<TemperatureChangedEventArgs> TemperatureChanged;


    public decimal Temperature
    {
        get => _temperature;
        set
        {
            if (_temperature != value)
            {
                _temperature = value;
                OnTemperatureChanged(new TemperatureChangedEventArgs(_temperature));
            }
        }
    }


    protected virtual void OnTemperatureChanged(TemperatureChangedEventArgs e)
    {
        TemperatureChanged?.Invoke(this, e); // Thread-safe invocation
    }
}


// Custom EventArgs
public class TemperatureChangedEventArgs : EventArgs
{
    public decimal NewTemperature { get; }
    public TemperatureChangedEventArgs(decimal temp) => NewTemperature = temp;
}
```


Ở phía subscriber:


```csharp
var sensor = new Thermometer();
sensor.TemperatureChanged += (sender, args) =>
    Console.WriteLine($"Temp is now: {args.NewTemperature}°C");
```


Ưu điểm so với cách thủ công:


· Không cần danh sách observer riêng, delegate multicast lo việc gọi.
· Đóng gói cao, chỉ subject được kích hoạt sự kiện.
· Chuẩn hóa chữ ký qua EventHandler và EventArgs, dễ dàng tích hợp với WinForms, ASP.NET, WPF.


3.2 Multicast delegate và thứ tự gọi


Một event có thể có nhiều handler gắn vào. Khi invoke, tất cả được gọi tuần tự theo thứ tự đăng ký. Nếu một handler ném ngoại lệ, các handler sau không được gọi. Có thể dùng Delegate.GetInvocationList() để xử lý riêng từng handler nếu cần.


3.3 Thread safety


Phép kiểm tra null TemperatureChanged?.Invoke(...) là thread-safe cho delegate (vì delegate là bất biến, dấu ?. tạo bản sao cục bộ). Tuy nhiên, trong môi trường đa luồng, cần lưu ý khi attach/detach đồng thời.


3.4 Memory Leaks và Weak Event Pattern


Observer bị subject giữ tham chiếu mạnh. Nếu quên unsubscribe, subject sẽ giữ observer sống mãi, dẫn đến rò rỉ bộ nhớ, đặc biệt khi subject sống lâu hơn observer (ví dụ: ViewModel đăng ký sự kiện của service toàn cục). Giải pháp:


· Luôn -= trong Dispose() hoặc khi không cần.
· Weak Event Pattern: Sử dụng WeakReference để subject không giữ observer cứng. .NET cung cấp WeakEventManager (WPF) hoặc triển khai thủ công qua WeakReference<Action>.
· CompositeDisposable hoặc các thư viện như Reactive Extensions giúp hủy đăng ký hàng loạt (subscription disposable).

4. IObserver<T> và IObservable<T> – Mẫu Observer chuẩn hóa cho dữ liệu tuần tự


Từ .NET 4.0, hai interface này được thiết kế để biểu diễn một nhà cung cấp luồng thông báo (push-based notification) với ngữ nghĩa rõ ràng về lỗi và kết thúc.


Cấu trúc:


```csharp
public interface IObservable<T>
{
    IDisposable Subscribe(IObserver<T> observer);
}


public interface IObserver<T>
{
    void OnNext(T value);      // Dữ liệu tiếp theo
    void OnError(Exception error); // Lỗi xảy ra
    void OnCompleted();          // Không còn dữ liệu
}
```


Tư tưởng: IObservable<T> không phải là một sự kiện đơn lẻ, mà là một luồng các giá trị theo thời gian (sequence). Nó đặc biệt phù hợp cho dữ liệu liên tục, bất đồng bộ và có thể kết thúc hoặc gặp lỗi.


4.1 Ví dụ triển khai thủ công


```csharp
public class DataStream : IObservable<int>
{
    private readonly List<IObserver<int>> _observers = new();


    public IDisposable Subscribe(IObserver<int> observer)
    {
        _observers.Add(observer);
        return new Unsubscriber(_observers, observer);
    }


    public void Publish(int value)
    {
        foreach (var obs in _observers)
            obs.OnNext(value);
    }


    public void End()
    {
        foreach (var obs in _observers)
            obs.OnCompleted();
        _observers.Clear();
    }


    private class Unsubscriber : IDisposable
    {
        private readonly List<IObserver<int>> _observers;
        private readonly IObserver<int> _observer;
        public Unsubscriber(List<IObserver<int>> observers, IObserver<int> observer) { ... }
        public void Dispose() => _observers.Remove(_observer);
    }
}
```


Phía tiêu thụ:


```csharp
var stream = new DataStream();
var subscription = stream.Subscribe(new MyObserver());
stream.Publish(10);
subscription.Dispose(); // Hủy quan sát
```


4.2 Sử dụng Subject<T> từ Rx.NET


Trong thực tế, không cần tự viết luồng; thư viện System.Reactive (Rx.NET) cung cấp Subject<T> giống như một event multicaster cho IObservable<T>.


```csharp
var subject = new Subject<int>();


subject.Subscribe(
    onNext: x => Console.WriteLine($"Received {x}"),
    onError: ex => Console.WriteLine($"Error: {ex.Message}"),
    onCompleted: () => Console.WriteLine("Done")
);


subject.OnNext(1);
subject.OnNext(2);
subject.OnCompleted(); // Kết thúc luồng
```


4.3 So sánh Event vs IObservable


Tiêu chí Event (C# keyword) IObservable<T>
Mục đích Sự kiện rời rạc, không có kết thúc Luồng dữ liệu tuần tự, có thể hoàn thành/lỗi
Ngữ nghĩa lỗi Không có cơ chế tự nhiên OnError rõ ràng
Hủy đăng ký -= Dispose trả về từ Subscribe
Khả năng biến đổi LINQ không áp dụng được trực tiếp LINQ operators (Rx): filter, map, merge...
Bất đồng bộ Phụ thuộc vào handler Hỗ trợ scheduling, async
Mức độ tiêu chuẩn Gắn chặt với ngôn ngữ Giao diện .NET, độc lập ngôn ngữ


Khi chỉ cần thông báo một sự kiện đơn giản, event là tự nhiên. Khi cần làm việc với chuỗi sự kiện phức tạp, lọc, kết hợp, xử lý lỗi và bất đồng bộ, IObservable cùng Rx là lựa chọn sáng suốt.

5. Các biến thể và kỹ thuật nâng cao


5.1 Event Aggregator / Event Bus


Khi có nhiều publisher và subscriber phân tán, việc kết nối trực tiếp dẫn đến coupling chằng chịt. Event Aggregator là một trung tâm điều phối sự kiện, các đối tượng chỉ biết đến bus, không biết nhau.


Trong C#, có thể tự xây dựng hoặc sử dụng các thư viện (Prism, Shiny, MediatR cho notification):


```csharp
public interface IEventAggregator
{
    void Subscribe<T>(Action<T> handler);
    void Publish<T>(T eventArgs);
}
```


5.2 Observer với async/await


Handler của sự kiện thường là void, không hỗ trợ await. Để xử lý bất đồng bộ, ta phải dùng async void nhưng cần kiểm soát cẩn thận ngoại lệ. Giải pháp là sử dụng AsyncEvent (tự viết hoặc dùng Rx).


Rx hỗ trợ async thông qua SubscribeAsync hoặc các toán tử như SelectMany kết hợp Task.


Ví dụ:


```csharp
subject
    .SelectMany(async x => {
        await Task.Delay(100);
        return x * 2;
    })
    .Subscribe(result => Console.WriteLine(result));
```


5.3 Weak Observer qua WeakReference


Trong ứng dụng WPF, PropertyChangedEventManager cho phép lắng nghe INotifyPropertyChanged.PropertyChanged mà không giữ tham chiếu mạnh. Tương tự, có thể tạo WeakObserver<T> wrap một IObserver<T> bên trong WeakReference.


5.4 Observer trong lập trình phản ứng (ReactiveUI, Blazor)


Blazor sử dụng EventCallback cho giao tiếp component; nó là một dạng Observer pattern. ReactiveUI xây dựng mọi thứ trên IObservable, cho phép data binding kiểu reactive.

6. Ứng dụng thực tế trong .NET


· Giao diện người dùng: WinForms button click, WPF command, Blazor EventCallback.
· INotifyPropertyChanged: Nền tảng cho data binding. Mỗi ViewModel là subject, View là observer.
· FileSystemWatcher: Là một subject phát sinh sự kiện khi file thay đổi.
· System.Threading.Timer: Callback delegate hoặc event.
· ASP.NET Core Middleware pipeline: Không phải Observer thuần túy nhưng triết lý “chuỗi handler” có nét tương đồng.
· SignalR: Hub bên server phát sự kiện xuống client, mỗi client là observer.


---


7. Lựa chọn triển khai phù hợp


Kịch bản Nên dùng Giải thích
Nhiều handler cần biết sự kiện đơn giản event + Delegate Đơn giản, tích hợp tốt, hiệu năng cao.
Cần hủy đăng ký an toàn, tránh rò rỉ IObservable + Rx hoặc WeakEvent Quản lý vòng đời subscription rõ ràng.
Chuỗi sự kiện cần lọc, biến đổi, kết hợp Rx.NET (IObservable<T>) LINQ operators mạnh mẽ, reactive.
Giao tiếp giữa các module độc lập Event Aggregator Giảm coupling, dễ kiểm thử.
Data binding trong WPF/MAUI INotifyPropertyChanged (event) Hỗ trợ bởi framework, tối ưu cho UI.

8. Lưu ý và best practices


1. Tránh null check trong event handler: Luôn dùng ?.Invoke để đảm bảo an toàn luồng.
2. Không quên hủy đăng ký: Nhất là khi observer có vòng đời ngắn hơn subject. Triển khai IDisposable cho observer và hủy đăng ký trong Dispose().
3. Sử dụng EventArgs chuẩn cho event: Kế thừa EventArgs, dùng EventHandler<T> thay vì delegate tùy chỉnh; điều này tạo nhất quán và tương thích.
4. Event không hỗ trợ async trực tiếp: Nếu handler cần async, bọc async void và log ngoại lệ bên trong.
5. Hạn chế số lượng observer trên một event: Khi số lượng lớn, hãy nghĩ đến Rx hoặc mô hình mediator để phân tải.
6. Kiểm tra đa luồng: Nếu event được raise từ background thread, observer cần đảm bảo cập nhật UI an toàn (dùng SynchronizationContext hoặc Dispatcher).
7. Sử dụng IObservable cho stream có kết thúc: Điều này giúp tài nguyên được giải phóng khi hoàn thành.
8. Chỉ nên dùng event cho hành vi “fire-and-forget” thông báo, không dùng cho yêu cầu-xác nhận (request-response).


---


9. Kết luận


Observer Pattern trong C# đã được nâng cấp từ một mẫu thiết kế thủ công thành một phần tự nhiên của ngôn ngữ và thư viện. Với delegate/event, lập trình viên có thể gắn kết các thành phần một cách lỏng lẻo, phục vụ hoàn hảo cho các sự kiện rời rạc. Khi cần xử lý luồng dữ liệu phức tạp, IObservable<T> và Reactive Extensions mở rộng mạnh mẽ pattern, cho phép lập trình phản ứng với các toán tử chuẩn hóa, khả năng kết hợp và kiểm soát lỗi tinh tế. Việc lựa chọn đúng công cụ và tuân thủ best practices sẽ giúp hệ thống mở rộng, dễ bảo trì và tránh các lỗi tiềm ẩn về bộ nhớ và luồng.
